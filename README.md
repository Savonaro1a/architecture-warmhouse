### 1. Описание функциональности монолитного приложения

**Управление отоплением:**

- Пользователи могут удаленно включать и отключать отопление в своих домах.
- Система поддерживает отправку команд устройствам, установленным в домах клиентов, обеспечивая контроль над обогревательными приборами.

**Мониторинг температуры:**

- Пользователи могут просматривать текущую температуру в своих домах через веб-интерфейс.
- Система регулярно запрашивает данные о температуре с установленных датчиков и передает их пользователю.

### 2. Анализ архитектуры монолитного приложения

*Язык программирования:* Go.
*СУБД:* PostgreSQL.
*Архитектура:* Монолитная, вся логика сосредоточена внутри единого исполняемого файла.
*Взаимодействие:* Синхронное выполнение всех операций.
*Масштабируемость:* Сложно масштабируется частями, требует перезапуска всей системы при изменениях.
*Развертывание:* Любое изменение влечет остановку всего приложения.

### 3. Определение доменов и границы контекстов

*Домен "Управление устройствами":* Отвечает за сбор данных с датчиков, отправку команд устройствам и обработку статуса оборудования.
*Домен "Интерфейсы пользователей":* Обеспечивает доступ пользователей к управлению системой через удобный интерфейс.

Эти домены образуют отдельные области ответственности и требуют дальнейшего разделения на микросервисы.

### 4. Проблемы монолитного решения

- Отсутствие гибкости: Любые изменения в одной части системы влияют на всю систему.
- Трудности масштабирования: Невозможно масштабировать отдельно одну часть системы (например, модуль мониторинга).
- Зависимости между компонентами: Все компоненты тесно связаны друг с другом, что затрудняет внесение изменений и устранение ошибок.
- Ограниченная поддержка: Развитие нового функционала сильно ограничено возможностями текущей структуры.
- Высокие риски при развертывании: Каждое обновление потенциально рискованно, так как влияет на стабильность всей системы.

# Задание 4. Создание и документирование API

### 1. Тип API

В данной спецификации я использовал REST API для взаимодействия между микросервисами.

Причины выбора REST API:

- Синхронность и простота реализации:

REST API идеально подходят для большинства синхронных операций, когда клиенту (другому микросервису) необходимо дождаться ответа на запрос.
Простота разработки и ясность методов (GET, POST, PUT, DELETE) делают REST API легкими для понимания и сопровождения.

- Стандартизация и совместимость:

REST API широко распространены и поддерживают большинство современных технологий и инструментов (например, Swagger, Postman).
Стандартные коды HTTP-статуса (200, 404, 500 и т.д.) позволяют прозрачно указывать на успехи и ошибки, что облегчает диагностику и устранение проблем.

- Расширяемость и гибкость:

REST API легко адаптируются к изменению структуры данных или появлению новых маршрутов. Новые пути можно добавлять постепенно, без кардинальных перестроек.
Возможность кастомизировать заголовки, тела запросов и ответов позволяет гибко решать специфические задачи.

- Масштабируемость и кеширование:

REST API поддерживают кэширование на уровнях прокси-серверов и CDN, что улучшает производительность при частых запросах.
Масштабирование на уровне сервисов достигается добавлением новых экземпляров REST API без значительных изменений архитектуры.

# Задание 5. Работа с docker и docker-compose

Перейдите в apps.

Там находится приложение-монолит для работы с датчиками температуры. В README.md описано как запустить решение.

Вам нужно:

1) сделать простое приложение temperature-api на любом удобном для вас языке программирования, которое при запросе /temperature?location= будет отдавать рандомное значение температуры.

Locations - название комнаты, sensorId - идентификатор названия комнаты

```
	// If no location is provided, use a default based on sensor ID
	if location == "" {
		switch sensorID {
		case "1":
			location = "Living Room"
		case "2":
			location = "Bedroom"
		case "3":
			location = "Kitchen"
		default:
			location = "Unknown"
		}
	}

	// If no sensor ID is provided, generate one based on location
	if sensorID == "" {
		switch location {
		case "Living Room":
			sensorID = "1"
		case "Bedroom":
			sensorID = "2"
		case "Kitchen":
			sensorID = "3"
		default:
			sensorID = "0"
		}
	}
```

2) Приложение следует упаковать в Docker и добавить в docker-compose. Порт по умолчанию должен быть 8081

3) Кроме того для smart_home приложения требуется база данных - добавьте в docker-compose файл настройки для запуска postgres с указанием скрипта инициализации ./smart_home/init.sql

Для проверки можно использовать Postman коллекцию smarthome-api.postman_collection.json и вызвать:

- Create Sensor
- Get All Sensors

Должно при каждом вызове отображаться разное значение температуры
